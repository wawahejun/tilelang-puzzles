# Copy Implementation Guide

This document provides specific implementation code and detailed technical explanations for Copy operations.

---

## Part 1: Serial Copy Implementation

### Complete Code

```python
@tilelang.jit
def tl_copy_1d_serial(A):
    # Declare constants and tensor shape/dtype
    N = T.const("N")
    A: T.Tensor((N,), T.float16)
    B = T.empty((N,), T.float16)

    # Launch kernel with one thread block
    with T.Kernel(1, threads=1) as _:
        T.copy(A, B)

    return B
```

### Code Explanation

| Code | Description |
|------|-------------|
| `@tilelang.jit` | JIT compiler decorator, compiles Python function to GPU kernel |
| `N = T.const("N")` | Declare compile-time constant, passed at runtime via dict `{"N": 1024}` |
| `A: T.Tensor((N,), T.float16)` | Declare input tensor shape and type |
| `B = T.empty((N,), T.float16)` | Allocate output tensor memory |
| `T.Kernel(1, threads=1)` | Launch 1 thread block with 1 thread per block |
| `T.copy(A, B)` | Whole tensor copy, no indexing |

**Note**: With only 1 thread, `T.copy` degenerates to serial loop copy, but still has vectorization optimization (e.g., 128-bit per copy).

## Part 2: Multi-thread Copy Implementation

### Complete Code

```python
@tilelang.jit
def tl_copy_1d_multi_threads(A):
    N = T.const("N")
    A: T.Tensor((N,), T.float16)
    B = T.empty((N,), T.float16)

    # Launch 1 thread block with 256 threads per block
    with T.Kernel(1, threads=256) as _:
        T.copy(A, B)

    return B
```

### Key Points

**Minimal Change**: The only difference from serial version is `threads=1` → `threads=256`.

`T.copy` internally automatically distributes N elements evenly among 256 threads:

```
Thread 0:   Process elements [0, N/256)
Thread 1:   Process elements [N/256, 2*N/256)
...
Thread 255: Process elements [255*N/256, N)
```

## Part 3: Multi-block Copy Implementation

### Complete Code

```python
@tilelang.jit
def tl_copy_1d_parallel(A, BLOCK_N: int):
    N = T.const("N")
    A: T.Tensor((N,), T.float16)
    B = T.empty((N,), T.float16)

    # Launch N // BLOCK_N thread blocks
    with T.Kernel(N // BLOCK_N, threads=256) as pid_n:
        # Use slice syntax, each thread block processes BLOCK_N elements
        T.copy(
            A[pid_n * BLOCK_N : (pid_n + 1) * BLOCK_N],
            B[pid_n * BLOCK_N : (pid_n + 1) * BLOCK_N],
        )

    return B
```

### Data Blocking Diagram

Assume `N = 1024*256`, `BLOCK_N = 1024`, then 256 thread blocks are launched:

```
Data:   [----BLOCK_N----|----BLOCK_N----|----BLOCK_N----|...]
            Block 0          Block 1          Block 2
            pid_n=0          pid_n=1          pid_n=2

Inside each Block:
Block 0:  T.copy(A[0:1024],       B[0:1024])        ← 256 threads parallel
Block 1:  T.copy(A[1024:2048],    B[1024:2048])     ← 256 threads parallel
Block 2:  T.copy(A[2048:3072],    B[2048:3072])     ← 256 threads parallel
...
```

### Key Code Analysis

| Code | Description |
|------|-------------|
| `N // BLOCK_N` | Number of thread blocks, each block processes `BLOCK_N` elements |
| `pid_n` | Thread block index, range `[0, N // BLOCK_N)` |
| `pid_n * BLOCK_N` | Data start position for current block |
| `(pid_n + 1) * BLOCK_N` | Data end position for current block |

## Comparison of Three Implementations

| Version | blocks | threads/block | Total parallelism | Use Case |
|---------|--------|---------------|-------------------|----------|
| Serial | 1 | 1 | 1 | Learning/Debugging |
| Multi-threads | 1 | 256 | 256 | Small data size |
| Parallel | N/BLOCK_N | 256 | 256×N/BLOCK_N | Large data size |

**Stepwise Performance Improvement**: From 1 thread → 256 threads → 256 blocks × 256 threads, fully utilizing GPU parallel capability.

## Performance Analysis

### Theoretical Throughput

Assume `N = 262,144`, `BLOCK_N = 1024`:

```
Serial version:
- 1 thread processes 262,144 elements
- Time ∝ N

Multi-thread version:
- 256 threads process 262,144 elements
- Each thread processes ~1,024 elements
- Time ∝ N/256

Multi-block version:
- 256 blocks × 256 threads = 65,536 threads
- Each thread processes ~4 elements
- Time ∝ N/65536
```

### Practical Performance Factors

1. **Memory bandwidth**: GPU copy is memory bandwidth-bound operation
2. **Memory coalescing**: Adjacent threads access adjacent addresses, improving bandwidth utilization
3. **Vectorization**: `T.copy` automatically uses 128-bit vectorized load/store

## Detailed Explanation of Two T.copy Indexing Methods

### Method 1: Specify Start Index Only (Auto-inference)

```python
# Target buffer size is (BLOCK_M, BLOCK_K)
A_local = T.alloc_fragment((BLOCK_M, BLOCK_K), dtype)
T.copy(A[pid_m * BLOCK_M, k * BLOCK_K], A_local)
```

**How it works**:
- TileLang infers range from target buffer shape
- Equivalent to `A[pid_m * BLOCK_M : pid_m * BLOCK_M + BLOCK_M, k * BLOCK_K : k * BLOCK_K + BLOCK_K]`

**Applicable Conditions**:
- ✅ Target is local buffer allocated by `T.alloc_fragment` or `T.alloc_shared`
- ❌ Target is global tensor (no shape information, cannot infer)

### Method 2: Explicit Slice

```python
T.copy(
    A[pid_n * BLOCK_N : (pid_n + 1) * BLOCK_N],
    B[pid_n * BLOCK_N : (pid_n + 1) * BLOCK_N],
)
```

**Use Cases**:
- Must use when both source and target are global tensors
- Need to explicitly specify range

### Common Errors

```python
# ❌ Error: Global tensor cannot use auto-inference
T.copy(A[bx * BLOCK_N], B[bx * BLOCK_N])

# ✅ Correct: Global tensor must use explicit slice
T.copy(
    A[bx * BLOCK_N : (bx + 1) * BLOCK_N],
    B[bx * BLOCK_N : (bx + 1) * BLOCK_N],
)
```

| Target Type | Can Auto-infer | Example |
|-------------|----------------|---------|
| `T.alloc_fragment(...)` | ✅ Yes | `T.copy(A[start], local_buf)` |
| `T.alloc_shared(...)` | ✅ Yes | `T.copy(A[start], shared_buf)` |
| Global tensor | ❌ No | `T.copy(A[start], B[start])` ❌ |

## Frequently Asked Questions

### Q1: Why doesn't `T.copy` need manual work distribution to threads?

`T.copy` internally implements automatic work distribution:
1. Calculate total number of elements
2. Distribute evenly based on `threads` count
3. Each thread processes a contiguous segment of data
4. Automatically use vectorized instructions (e.g., `ld.global.v4.f16`)

### Q2: What should the `threads` parameter be set to?

| threads | Description |
|---------|-------------|
| 128 | Balanced choice, suitable for most scenarios |
| 256 | Higher parallelism, suitable for large data sizes |
| 512+ | May cause register pressure, needs testing |

### Q3: Why does the multi-block version need slice syntax?

Because each block needs to process different data segments, the range must be explicitly specified. The first two versions have all blocks processing the same data (only one block), so slicing isn't needed.

---

## Summary

Three key points:

1. **`T.copy` is automatically parallel within a block** — You just tell it what to copy, it handles thread distribution itself.
2. **Multi-block requires manual slicing** — Use `pid_n` (block index) to calculate the data range each block is responsible for.
3. **Stepwise performance improvement** — From 1 thread → 256 threads → multiple blocks × 256 threads, fully utilizing GPU parallel capability.
