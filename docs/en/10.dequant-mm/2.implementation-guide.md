# Dequantized Matrix Multiplication Implementation Guide

## Overview

This document explains in detail how to implement simple FP16×INT4 dequantized matrix multiplication operators, including packing format, dequantization, and matrix multiplication optimization strategies.

> Complete code in this document assumes `M % BLOCK_M == 0`, `N % BLOCK_N == 0`, `K % BLOCK_K == 0`.
> Although kernel grid uses `T.ceildiv`, without accompanying mask/padding, the last tile still needs separate handling.

## Core Constraints

Key characteristics of dequantized matrix multiplication:

- **Packed Storage**: Two INT4s packed in one UINT8
- **Bit Operations**: Need bitwise operations to extract INT4 values
- **Offset Restoration**: Current teaching version maps nibble `[0, 15]` back to `[-8, 7]`
- **Memory Efficiency**: Weight memory usage reduced by 75%

---

## Part 1: INT4 Packing Format

### Packing Principle

Each UINT8 byte stores two INT4 values:

```
Byte Layout:
+--------+--------+
| High 4 | Low 4  |
| B_high | B_low  |
+--------+--------+
   7:4      3:0

Example:
Original values: B_low=5, B_high=-2
Signed INT4: 5 → unsigned 13, -2 → unsigned 6
Packed: 6 << 4 | 13 = 0x6D = 109

Unpack:
B_low  = 0x6D & 0x0F = 13 → 13 - 8 = 5
B_high = (0x6D >> 4) & 0x0F = 6 → 6 - 8 = -2
```

### Numerical Restoration in Teaching Version

| nibble value | Restored value | Binary |
|-------------|----------------|--------|
| 0 | -8 | 0000 |
| 1 | -7 | 0001 |
| ... | ... | ... |
| 7 | -1 | 0111 |
| 8 | 0 | 1000 |
| 9 | 1 | 1001 |
| ... | ... | ... |
| 15 | 7 | 1111 |

This table describes the **restoration rule used in current puzzle**, not a universal definition for all INT4 quantization formats.
In more general quantized inference, usually need to multiply by `scale` again, sometimes handle explicit `zero_point`.

## Part 2: Dequantized Matrix Multiplication Implementation

### Implementation Approach

1. **Blocking Strategy**: Block in M and N dimensions, pipeline in K dimension
2. **Dequantization**: On-the-fly unpack and simplest offset restoration in shared memory
3. **Matrix Multiplication**: Use T.gemm to leverage Tensor Core

### Complete Implementation

```python
@tilelang.jit
def tl_dequant_matmul(A, B, BLOCK_M: int, BLOCK_N: int, BLOCK_K: int):
    M, N, K = T.const("M, N, K")
    A_dtype = T.float16
    B_storage_dtype = T.uint8
    A: T.Tensor((M, K), A_dtype)
    B: T.Tensor((K, N // 2), B_storage_dtype)
    C = T.empty((M, N), A_dtype)
    accum_dtype = T.float32

    with T.Kernel(T.ceildiv(M, BLOCK_M), T.ceildiv(N, BLOCK_N), threads=128) as (
        pid_m,
        pid_n,
    ):
        # Allocate memory
        A_shared = T.alloc_shared((BLOCK_M, BLOCK_K), A_dtype)
        B_shared = T.alloc_shared((BLOCK_K, BLOCK_N // 2), B_storage_dtype)
        B_dequantized = T.alloc_shared((BLOCK_K, BLOCK_N), A_dtype)  # Unpacked B
        C_local = T.alloc_fragment((BLOCK_M, BLOCK_N), accum_dtype)

        # Clear accumulator
        T.clear(C_local)

        # K dimension pipeline
        for k in T.Pipelined(T.ceildiv(K, BLOCK_K), num_stages=3):
            # Load A and B
            T.copy(A[pid_m * BLOCK_M, k * BLOCK_K], A_shared)
            T.copy(B[k * BLOCK_K, pid_n * BLOCK_N // 2], B_shared)

            # Dequantization: Unpack INT4 and convert
            for i, j in T.Parallel(BLOCK_K, BLOCK_N // 2):
                # Low 4 bits → even columns
                B_dequantized[i, j * 2] = T.cast(B_shared[i, j] & 0x0F, A_dtype) - 8.0
                # High 4 bits → odd columns
                B_dequantized[i, j * 2 + 1] = T.cast((B_shared[i, j] >> 4) & 0x0F, A_dtype) - 8.0

            # Matrix multiplication
            T.gemm(A_shared, B_dequantized, C_local)

        # Write back result
        T.copy(C_local, C[pid_m * BLOCK_M, pid_n * BLOCK_N])

    return C
```

## Part 3: Key Techniques Explained

### 1. Bit Operation Implementation

```python
# Extract low 4 bits (B_low)
B_low = B_shared[i, j] & 0x0F
# 0x0F = 00001111 (binary)
# Example: 0xAB & 0x0F = 0x0B = 11

# Extract high 4 bits (B_high)
B_high = (B_shared[i, j] >> 4) & 0x0F
# Example: (0xAB >> 4) = 0x0A = 10
```

### 2. Type Conversion and Offset

```python
# Teaching version restoration in TileLang
B_dequantized[i, j * 2] = T.cast(B_shared[i, j] & 0x0F, A_dtype) - 8.0
B_dequantized[i, j * 2 + 1] = T.cast((B_shared[i, j] >> 4) & 0x0F, A_dtype) - 8.0

# Decomposed steps:
# 1. B_shared[i, j] & 0x0F: Extract low 4 bits (unsigned, range 0-15)
# 2. T.cast(..., A_dtype): Convert to float16
# 3. - 8.0: Use fixed offset to restore nibble to [-8, 7]
```

If extending to real quantization format, this step usually becomes:

```python
real_weight = scale * (q - zero_point)
```

Current puzzle is equivalent to:

```python
scale = 1
zero_point = 8
real_weight = q - 8
```

### 3. Memory Layout

```python
# Packed B: [K, N // 2] (UINT8)
# After unpacking: [K, N] (FP16)

# Index correspondence:
# B[k, j] low 4 bits → B_dequantized[k, j * 2]
# B[k, j] high 4 bits → B_dequantized[k, j * 2 + 1]

# Example (N=4, after packing N//2=2):
# B[0, 0] low 4 bits → B_dequantized[0, 0]
# B[0, 0] high 4 bits → B_dequantized[0, 1]
# B[0, 1] low 4 bits → B_dequantized[0, 2]
# B[0, 1] high 4 bits → B_dequantized[0, 3]
```

### 4. Software Pipeline

```python
# Use T.Pipelined to overlap data loading and computation
for k in T.Pipelined(T.ceildiv(K, BLOCK_K), num_stages=3):
    # Stage 1: Load data for next tile
    # Stage 2: Dequantize current tile
    # Stage 3: Compute matrix multiplication

    # 3-stage pipeline can significantly hide latency of loading, unpacking, and subsequent gemm
```

This is also an important approach when later extending to `scale`/`zero_point` metadata:
Even when dequantization logic becomes complex, still want to overlap "read metadata + decode + gemm" as much as possible.

## Part 4: Performance Optimization Strategies

### 1. Shared Memory Usage

```python
# Calculate shared memory requirements
A_shared_size = BLOCK_M * BLOCK_K * 2  # FP16 = 2 bytes
B_shared_size = BLOCK_K * (BLOCK_N // 2) * 1  # UINT8 = 1 byte
B_dequantized_size = BLOCK_K * BLOCK_N * 2  # FP16 = 2 bytes

# Example: BLOCK_M=128, BLOCK_N=128, BLOCK_K=64
# A_shared = 128 * 64 * 2 = 16 KB
# B_shared = 64 * 64 * 1 = 4 KB
# B_dequantized = 64 * 128 * 2 = 16 KB
# Total: 36 KB (A100 shared memory ~164 KB)
```

### 2. Block Size Selection

```python
# Recommended configuration
BLOCK_M = 128  # M dimension block size
BLOCK_N = 128  # N dimension block size
BLOCK_K = 64   # K dimension block size

# Considerations:
# 1. Shared memory limits
# 2. Tensor Core efficiency (multiples of 16)
# 3. Parallelism (threads=128)
```

### 2.5 From Answer Version to Advanced Version

Current implementation does:

```python
uint8 -> unpack two int4 -> q - 8 -> gemm
```

More advanced versions usually continue adding:

```python
uint8 -> unpack -> q -> (q - zero_point) * scale -> gemm
```

Common engineering variations include:
- `scale` stored per-column (per-channel)
- `scale` stored per-group (group-wise)
- `zero_point` explicitly stored, not hardcoded to 8
- Dequantized results don't land in full shared memory, but more fine-grained feeding to MMA

### 3. Pipeline Depth

```python
# num_stages selection
num_stages = 3  # Recommend 2-4

# Too few: Cannot hide memory latency
# Too many: Increase shared memory pressure
```

## Part 5: Differences from Standard GEMM

### Memory Access Pattern

| Feature | Standard GEMM | Dequantized GEMM |
|---------|--------------|------------------|
| A Matrix Load | Direct load | Direct load |
| B Matrix Load | Direct load | Load + Unpack + Dequantize |
| Compute Core | T.gemm | T.gemm (same) |
| Extra Overhead | None | Dequantization computation |

### Data Flow

```
Standard GEMM:
A (HBM) → A_shared → T.gemm → C_local → C (HBM)
B (HBM) → B_shared → T.gemm → C_local → C (HBM)

Dequantized GEMM:
A (HBM) → A_shared → T.gemm → C_local → C (HBM)
B (HBM) → B_shared → B_dequantized → T.gemm
                      ↑
                   Unpack + teaching version restoration
```

If extending to real quantization kernels, this data flow usually also inserts `scale/zero_point` reading and broadcasting.
