# Puzzle 10: Dequantized Matrix Multiplication Code Analysis

## Overview

This puzzle introduces a key technique in deep learning model deployment: **Dequantized Matrix Multiplication**.
For teaching clarity, this document starts with the simplest version: packing two 4-bit weights into one byte, unpacking and applying a fixed offset dequantization, then performing regular matrix multiplication.

## Why Is Dequantized Matrix Multiplication So Important?

In LLM deployment, model quantization is a key technique to reduce memory usage and accelerate inference:

- **Memory Savings**: INT4 quantization uses 75% less VRAM than FP16
- **Bandwidth Optimization**: Smaller data transfer, reducing memory bandwidth pressure
- **Inference Acceleration**: Combined with Tensor Core, enables efficient inference
- **Practical Applications**: Core operator in GPTQ, AWQ, llama.cpp and other quantization frameworks

**Typical Application Scenarios**:
- GPTQ quantized model inference
- AWQ quantized model inference
- llama.cpp INT4 quantization
- vLLM quantized inference

> Note: Real-world quantization kernels often have `scale`, `zero-point`, group-wise/per-channel metadata, and deeper fusion of dequantization and matrix multiplication.
> This puzzle aligns with the "most basic packed-int4 unpack + offset restoration + GEMM" teaching version, not a complete reproduction of production implementations from these frameworks.

## Problem Definition

### Quantization Background

**INT4 Quantization**: Quantize FP16 weights to 4-bit integers, but 4-bit is smaller than a byte, usually packed for storage.

For step-by-step learning, this document splits "how to restore quantized weights" into three levels:

1. **Unpack**
   Extract high 4 bits and low 4 bits from a `uint8`.
2. **Sign/Offset Restoration**
   Map 0-15 nibble back to a small integer range centered around 0.
3. **True Dequantization (advanced)**
   In real quantization systems, usually need to multiply by `scale`, sometimes subtract `zero_point`, i.e.
   `real_value = scale * (q - zero_point)`.

Current puzzle only implements to level 2, equivalent to a very simple special case: `scale = 1`, `zero_point = 8`.

```
Original Weights (FP16): [w0, w1, w2, w3, ...]  Each element 16 bits
Quantized (INT4):        [w0, w1, w2, w3, ...]  Each element 4 bits
Packed Storage (UINT8):  [w0|w1, w2|w3, ...]    Each byte stores 2 INT4s
```

### Input and Output

**Input**:
- `A: [M, K]` - Matrix A (float16), activations
- `B: [K, N // 2]` - Matrix B (uint8), packed INT4 weights
- `M: int` - Number of rows in matrix A, 1 ≤ M ≤ 8192
- `N: int` - Actual columns in matrix B, 1 ≤ N ≤ 8192
- `K: int` - K dimension, 1 ≤ K ≤ 8192

**Output**:
- `C: [M, N]` - Output matrix (float16)

**Intermediate Variables**:
- `ACC0, ACC1: float32` - Accumulators
- `B_high, B_low: float16` - Unpacked INT4 values

> Current puzzle's reference implementation focuses on the main logic of "unpack + restore + GEMM".
> For simplicity, assumes `M % BLOCK_M == 0`, `N % BLOCK_N == 0`, `K % BLOCK_K == 0`; for arbitrary sizes, need additional boundary handling for loading and write-back.

### Computation Definition

```python
for i in range(M):
    for j in range(N // 2):
        ACC0 = 0
        ACC1 = 0
        for k in range(K):
            # Unpack INT4 values
            B_low = float16(B[k, j] & 0x0F) - 8.0   # Low 4 bits
            B_high = float16((B[k, j] >> 4) & 0x0F) - 8.0  # High 4 bits
            # Accumulate computation
            ACC0 += A[i, k] * B_low
            ACC1 += A[i, k] * B_high
        C[i, j * 2] = ACC0
        C[i, j * 2 + 1] = ACC1
```

## Core Concepts

### 1. INT4 Packing Format

Each UINT8 stores two INT4 values:

```
UINT8 byte: [7:4] | [3:0]
              High    Low
           B_high  B_low

Example:
B[k, j] = 0xAB  (binary: 10101011)
B_low  = 0xAB & 0x0F = 0x0B = 11
B_high = (0xAB >> 4) & 0x0F = 0x0A = 10
```

### 2. Offset Restoration to Signed Range

Current puzzle's nibble value range is `[0, 15]`, by subtracting 8, restore to `[-8, 7]`:

```python
# nibble: [0, 15]
# After restoration: [-8, 7]

# Current puzzle's restoration formula
decoded_value = nibble - 8

# Examples
0x0 = 0  → 0 - 8 = -8
0x7 = 7  → 7 - 8 = -1
0x8 = 8  → 8 - 8 = 0
0xF = 15 → 15 - 8 = 7
```

This step is intuitive enough for teaching, but note it doesn't cover all INT4 encoding methods.
For example, in real quantization frameworks, commonly combined with `scale`/`zero_point` for affine dequantization, not just a fixed `-8`.

### 3. Bitwise Operation Details

```python
# Extract low 4 bits
B_low = B[k, j] & 0x0F
# 0x0F = 00001111 (binary)
# AND operation preserves low 4 bits, clears high 4 bits

# Extract high 4 bits
B_high = (B[k, j] >> 4) & 0x0F
# First right shift 4 bits, then AND with 0x0F ensures only 4 bits
```

## PyTorch Reference Implementation

```python
def ref_dequant_matmul(A: torch.Tensor, B: torch.Tensor):
    assert len(A.shape) == 2
    assert len(B.shape) == 2
    assert A.shape[1] == B.shape[0]  # K
    assert A.dtype == torch.float16
    assert B.dtype == torch.uint8

    K = A.shape[1]
    N = B.shape[1] * 2  # B.shape[1] == N // 2 because of packing

    # Teaching version: First unpack, then do simplest offset restoration
    B_dequantized = torch.zeros((K, N), dtype=torch.float16, device=B.device)
    B_dequantized[:, ::2] = B[:, :] & 0x0F      # Even columns: low 4 bits
    B_dequantized[:, 1::2] = (B[:, :] >> 4) & 0x0F  # Odd columns: high 4 bits
    B_dequantized = B_dequantized.to(torch.float16) - 8.0  # Offset restore to [-8, 7]

    # Matrix multiplication
    return torch.matmul(input=A, other=B_dequantized)
```

## Comparison with Standard Matrix Multiplication

| Feature | Standard GEMM (FP16×FP16) | Dequantized GEMM (FP16×INT4) |
|---------|---------------------------|------------------------------|
| Weight Storage | FP16 (16 bits) | UINT8 (8 bits, packed 2 INT4s) |
| Weight Memory | N×K×2 bytes | N×K/2 bytes (75% savings) |
| Computation Steps | Direct matrix multiplication | Unpack → Dequantize → Matrix multiplication |
| Extra Overhead | None | Dequantization computation |
| Use Cases | Training, inference | Quantized model inference |

## From Teaching Version to Real Quantization Kernel

Current puzzle can be seen as the simplest special case of the following more general formula:

```python
real_weight = scale * (q - zero_point)
```

In this document:
- `q` comes from unpacked 4-bit value
- `zero_point = 8`
- `scale = 1`

If continuing toward engineering implementation, common next steps are:
- Add `scale` for each channel or each group
- Make `zero_point` also per-group metadata
- Fuse these metadata and unpack logic into matmul main loop

That is, this kernel is not the endpoint, but the first step to understand real GPTQ/AWQ class kernels.
