# Copy 实现指南

本文档提供 Copy 操作的具体实现代码和关键技术详解。

---

## Part 1: 串行 Copy 实现

### 完整代码

```python
@tilelang.jit
def tl_copy_1d_serial(A):
    # 声明常量和张量形状/数据类型
    N = T.const("N")
    A: T.Tensor((N,), T.float16)
    B = T.empty((N,), T.float16)

    # 启动一个线程块的内核
    with T.Kernel(1, threads=1) as _:
        T.copy(A, B)

    return B
```

### 代码详解

| 代码 | 说明 |
|------|------|
| `@tilelang.jit` | JIT 编译装饰器，将 Python 函数编译为 GPU 内核 |
| `N = T.const("N")` | 声明编译时常量，运行时通过字典 `{"N": 1024}` 传入 |
| `A: T.Tensor((N,), T.float16)` | 声明输入张量的形状和类型 |
| `B = T.empty((N,), T.float16)` | 分配输出张量内存 |
| `T.Kernel(1, threads=1)` | 启动 1 个线程块，每块 1 个线程 |
| `T.copy(A, B)` | 整张量拷贝，无索引 |

**注意**：这里只有 1 个线程，`T.copy` 会退化为串行循环拷贝，但仍然会有向量化优化（如 128-bit 每次拷贝）。

## Part 2: 多线程 Copy 实现

### 完整代码

```python
@tilelang.jit
def tl_copy_1d_multi_threads(A):
    N = T.const("N")
    A: T.Tensor((N,), T.float16)
    B = T.empty((N,), T.float16)

    # 启动 1 个线程块，每块 256 个线程
    with T.Kernel(1, threads=256) as _:
        T.copy(A, B)

    return B
```

### 关键点

**改动极小**：和串行版本唯一的区别就是 `threads=1` → `threads=256`。

`T.copy` 内部会自动把 N 个元素均匀分配给 256 个线程：

```
Thread 0:   处理元素 [0, N/256)
Thread 1:   处理元素 [N/256, 2*N/256)
...
Thread 255: 处理元素 [255*N/256, N)
```

## Part 3: 多线程块 Copy 实现

### 完整代码

```python
@tilelang.jit
def tl_copy_1d_parallel(A, BLOCK_N: int):
    N = T.const("N")
    A: T.Tensor((N,), T.float16)
    B = T.empty((N,), T.float16)

    # 启动 N // BLOCK_N 个线程块
    with T.Kernel(N // BLOCK_N, threads=256) as pid_n:
        # 使用切片语法，每个线程块处理 BLOCK_N 个元素
        T.copy(
            A[pid_n * BLOCK_N : (pid_n + 1) * BLOCK_N],
            B[pid_n * BLOCK_N : (pid_n + 1) * BLOCK_N],
        )

    return B
```

### 数据分块图解

假设 `N = 1024*256`，`BLOCK_N = 1024`，则启动 256 个线程块：

```
数据:   [----BLOCK_N----|----BLOCK_N----|----BLOCK_N----|...]
            Block 0          Block 1          Block 2
            pid_n=0          pid_n=1          pid_n=2

每个 Block 内部:
Block 0:  T.copy(A[0:1024],       B[0:1024])        ← 256 threads 并行
Block 1:  T.copy(A[1024:2048],    B[1024:2048])     ← 256 threads 并行
Block 2:  T.copy(A[2048:3072],    B[2048:3072])     ← 256 threads 并行
...
```

### 关键代码解析

| 代码 | 说明 |
|------|------|
| `N // BLOCK_N` | 线程块数量，每个块处理 `BLOCK_N` 个元素 |
| `pid_n` | 线程块索引，范围 `[0, N // BLOCK_N)` |
| `pid_n * BLOCK_N` | 当前块处理的数据起始位置 |
| `(pid_n + 1) * BLOCK_N` | 当前块处理的数据结束位置 |

## 三种实现对比

| 版本 | blocks | threads/block | 总并行度 | 适用场景 |
|------|--------|---------------|----------|----------|
| Serial | 1 | 1 | 1 | 学习/调试 |
| Multi-threads | 1 | 256 | 256 | 小数据量 |
| Parallel | N/BLOCK_N | 256 | 256×N/BLOCK_N | 大数据量 |

**性能阶梯式提升**：从 1 个线程 → 256 线程 → 256 个 block × 256 线程，充分利用 GPU 的并行能力。

## 性能分析

### 理论吞吐量

假设 `N = 262,144`，`BLOCK_N = 1024`：

```
串行版本:
- 1 个线程处理 262,144 个元素
- 耗时 ∝ N

多线程版本:
- 256 个线程处理 262,144 个元素
- 每个线程处理 ~1,024 个元素
- 耗时 ∝ N/256

多线程块版本:
- 256 个 block × 256 threads = 65,536 个线程
- 每个线程处理 ~4 个元素
- 耗时 ∝ N/65536
```

### 实际性能因素

1. **内存带宽**: GPU 拷贝是内存带宽受限操作
2. **内存合并**: 相邻线程访问相邻地址，提高带宽利用率
3. **向量化**: `T.copy` 自动使用 128-bit 向量化加载/存储

## `T.copy` 两种索引方式详解

### 方式 1：只指定起始索引（自动推断）

```python
# 目标 buffer 大小为 (BLOCK_M, BLOCK_K)
A_local = T.alloc_fragment((BLOCK_M, BLOCK_K), dtype)
T.copy(A[pid_m * BLOCK_M, k * BLOCK_K], A_local)
```

**工作原理**：
- TileLang 根据目标 buffer 的形状自动推断范围
- 等价于 `A[pid_m * BLOCK_M : pid_m * BLOCK_M + BLOCK_M, k * BLOCK_K : k * BLOCK_K + BLOCK_K]`

**适用条件**：
- ✅ 目标是 `T.alloc_fragment` 或 `T.alloc_shared` 分配的局部 buffer
- ❌ 目标是全局张量（没有形状信息，无法推断）

### 方式 2：显式切片

```python
T.copy(
    A[pid_n * BLOCK_N : (pid_n + 1) * BLOCK_N],
    B[pid_n * BLOCK_N : (pid_n + 1) * BLOCK_N],
)
```

**适用场景**：
- 源和目标都是全局张量时必须使用
- 需要明确指定范围

### 常见错误

```python
# ❌ 错误：全局张量不能用自动推断
T.copy(A[bx * BLOCK_N], B[bx * BLOCK_N])

# ✅ 正确：全局张量必须显式切片
T.copy(
    A[bx * BLOCK_N : (bx + 1) * BLOCK_N],
    B[bx * BLOCK_N : (bx + 1) * BLOCK_N],
)
```

| 目标类型 | 能否自动推断 | 示例 |
|----------|-------------|------|
| `T.alloc_fragment(...)` | ✅ 可以 | `T.copy(A[start], local_buf)` |
| `T.alloc_shared(...)` | ✅ 可以 | `T.copy(A[start], shared_buf)` |
| 全局张量 | ❌ 不能 | `T.copy(A[start], B[start])` ❌ |

## 常见问题

### Q1: 为什么 `T.copy` 不需要手动分配工作给线程？

`T.copy` 内部实现了自动工作分配：
1. 计算总元素数量
2. 根据 `threads` 数量均匀分配
3. 每个线程处理连续的一段数据
4. 自动使用向量化指令（如 `ld.global.v4.f16`）

### Q2: `threads` 参数应该设多少？

| threads | 说明 |
|---------|------|
| 128 | 平衡选择，适合大多数场景 |
| 256 | 更高并行度，适合数据量大时 |
| 512+ | 可能导致寄存器压力，需要测试 |

### Q3: 为什么多 block 版本要用切片语法？

因为每个 block 需要处理不同的数据段，必须显式指定范围。而前两个版本所有 block 处理相同的数据（只有一个 block），所以不需要切片。

---

## 总结

三个关键点：

1. **`T.copy` 是 block 内自动并行的** — 你只需要告诉它拷贝什么，线程分配它自己搞定。
2. **多 block 需要手动切片** — 用 `pid_n`（block index）计算每个 block 负责的数据范围。
3. **性能阶梯式提升** — 从 1 个线程 → 256 线程 → 多个 block × 256 线程，充分利用 GPU 的并行能力。
