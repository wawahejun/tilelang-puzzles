# Dequantized Matrix Multiplication 实现指南

## 概述

本文档详细说明如何实现简单的 FP16×INT4 反量化矩阵乘法算子，包括打包格式、反量化和矩阵乘法的优化策略。

> 本文展示的完整代码默认 `M % BLOCK_M == 0`、`N % BLOCK_N == 0`、`K % BLOCK_K == 0`。
> 虽然 kernel grid 使用了 `T.ceildiv`，但如果没有配套的 mask / padding，最后一个 tile 仍然需要单独处理。

## 核心约束

反量化矩阵乘法的关键特点是：

- **打包存储**: 两个 INT4 打包在一个 UINT8 中
- **位操作**: 需要按位操作提取 INT4 值
- **偏移还原**: 当前教学版本把 nibble `[0, 15]` 映射回 `[-8, 7]`
- **内存效率**: 权重内存占用减少 75%

## Part 1: INT4 打包格式

### 打包原理

每个 UINT8 字节存储两个 INT4 值：

```
字节布局:
+--------+--------+
| 高4位  | 低4位  |
| B_high | B_low  |
+--------+--------+
   7:4      3:0

示例:
原始值: B_low=5, B_high=-2
有符号 INT4: 5 → 无符号 13, -2 → 无符号 6
打包: 6 << 4 | 13 = 0x6D = 109

解包:
B_low  = 0x6D & 0x0F = 13 → 13 - 8 = 5
B_high = (0x6D >> 4) & 0x0F = 6 → 6 - 8 = -2
```

### 教学版中的数值还原

| nibble 值 | 还原值 | 二进制 |
|---------|---------|--------|
| 0 | -8 | 0000 |
| 1 | -7 | 0001 |
| ... | ... | ... |
| 7 | -1 | 0111 |
| 8 | 0 | 1000 |
| 9 | 1 | 1001 |
| ... | ... | ... |
| 15 | 7 | 1111 |

这张表描述的是**当前 puzzle 使用的还原规则**，不是所有 INT4 量化格式的统一定义。
更通用的量化推理里，通常还要再乘 `scale`，有时还要处理显式的 `zero_point`。

## Part 2: 反量化矩阵乘法实现

### 实现思路

1. **分块策略**: 在 M 和 N 维度分块，K 维度流水线
2. **反量化**: 在共享内存中即时解包和做最简单的偏移还原
3. **矩阵乘法**: 使用 T.gemm 利用 Tensor Core

### 完整实现

```python
@tilelang.jit
def tl_dequant_matmul(A, B, BLOCK_M: int, BLOCK_N: int, BLOCK_K: int):
    M, N, K = T.const("M, N, K")
    A_dtype = T.float16
    B_storage_dtype = T.uint8
    A: T.Tensor((M, K), A_dtype)
    B: T.Tensor((K, N // 2), B_storage_dtype)
    C = T.empty((M, N), A_dtype)
    accum_dtype = T.float32

    with T.Kernel(T.ceildiv(M, BLOCK_M), T.ceildiv(N, BLOCK_N), threads=128) as (
        pid_m,
        pid_n,
    ):
        # 分配内存
        A_shared = T.alloc_shared((BLOCK_M, BLOCK_K), A_dtype)
        B_shared = T.alloc_shared((BLOCK_K, BLOCK_N // 2), B_storage_dtype)
        B_dequantized = T.alloc_shared((BLOCK_K, BLOCK_N), A_dtype)  # 解包后的 B
        C_local = T.alloc_fragment((BLOCK_M, BLOCK_N), accum_dtype)

        # 清零累加器
        T.clear(C_local)

        # K 维度流水线
        for k in T.Pipelined(T.ceildiv(K, BLOCK_K), num_stages=3):
            # 加载 A 和 B
            T.copy(A[pid_m * BLOCK_M, k * BLOCK_K], A_shared)
            T.copy(B[k * BLOCK_K, pid_n * BLOCK_N // 2], B_shared)

            # 反量化: 解包 INT4 并转换
            for i, j in T.Parallel(BLOCK_K, BLOCK_N // 2):
                # 低 4 位 → 偶数列
                B_dequantized[i, j * 2] = T.cast(B_shared[i, j] & 0x0F, A_dtype) - 8.0
                # 高 4 位 → 奇数列
                B_dequantized[i, j * 2 + 1] = T.cast((B_shared[i, j] >> 4) & 0x0F, A_dtype) - 8.0

            # 矩阵乘法
            T.gemm(A_shared, B_dequantized, C_local)

        # 写回结果
        T.copy(C_local, C[pid_m * BLOCK_M, pid_n * BLOCK_N])

    return C
```

## Part 3: 关键技术详解

### 1. 位操作实现

```python
# 提取低 4 位 (B_low)
B_low = B_shared[i, j] & 0x0F
# 0x0F = 00001111 (二进制)
# 示例: 0xAB & 0x0F = 0x0B = 11

# 提取高 4 位 (B_high)
B_high = (B_shared[i, j] >> 4) & 0x0F
# 示例: (0xAB >> 4) = 0x0A = 10
```

### 2. 类型转换和偏移

```python
# TileLang 中的教学版还原
B_dequantized[i, j * 2] = T.cast(B_shared[i, j] & 0x0F, A_dtype) - 8.0
B_dequantized[i, j * 2 + 1] = T.cast((B_shared[i, j] >> 4) & 0x0F, A_dtype) - 8.0

# 分解步骤:
# 1. B_shared[i, j] & 0x0F: 提取低 4 位 (无符号, 范围 0-15)
# 2. T.cast(..., A_dtype): 转换为 float16
# 3. - 8.0: 用固定偏移把 nibble 还原到 [-8, 7]
```

如果进一步推广到真实量化格式，这一步通常会变成：

```python
real_weight = scale * (q - zero_point)
```

当前 puzzle 相当于：

```python
scale = 1
zero_point = 8
real_weight = q - 8
```

### 3. 内存布局

```python
# 打包的 B: [K, N // 2] (UINT8)
# 解包后:   [K, N] (FP16)

# 索引对应:
# B[k, j] 的低 4 位 → B_dequantized[k, j * 2]
# B[k, j] 的高 4 位 → B_dequantized[k, j * 2 + 1]

# 示例 (N=4, 打包后 N//2=2):
# B[0, 0] 低4位 → B_dequantized[0, 0]
# B[0, 0] 高4位 → B_dequantized[0, 1]
# B[0, 1] 低4位 → B_dequantized[0, 2]
# B[0, 1] 高4位 → B_dequantized[0, 3]
```

### 4. 软件流水线

```python
# 使用 T.Pipelined 重叠数据加载和计算
for k in T.Pipelined(T.ceildiv(K, BLOCK_K), num_stages=3):
    # 阶段 1: 加载下一个 tile 的数据
    # 阶段 2: 反量化当前 tile
    # 阶段 3: 计算矩阵乘法

    # 3 级流水线可以显著隐藏加载、解包和后续 gemm 的延迟
```

这也是后续扩展到 `scale`/`zero_point` 元数据时很重要的思路：
即使反量化逻辑变复杂，仍然希望把“读入 metadata + 解码 + gemm”尽量重叠起来。

## Part 4: 性能优化策略

### 1. 共享内存使用

```python
# 计算共享内存需求
A_shared_size = BLOCK_M * BLOCK_K * 2  # FP16 = 2 bytes
B_shared_size = BLOCK_K * (BLOCK_N // 2) * 1  # UINT8 = 1 byte
B_dequantized_size = BLOCK_K * BLOCK_N * 2  # FP16 = 2 bytes

# 示例: BLOCK_M=128, BLOCK_N=128, BLOCK_K=64
# A_shared = 128 * 64 * 2 = 16 KB
# B_shared = 64 * 64 * 1 = 4 KB
# B_dequantized = 64 * 128 * 2 = 16 KB
# 总计: 36 KB (A100 共享内存约 164 KB)
```

### 2. 分块大小选择

```python
# 推荐配置
BLOCK_M = 128  # M 维度块大小
BLOCK_N = 128  # N 维度块大小
BLOCK_K = 64   # K 维度块大小

# 考虑因素:
# 1. 共享内存限制
# 2. Tensor Core 效率 (16 的倍数)
# 3. 并行度 (threads=128)
```

### 2.5 从答案版到进阶版

当前做的是：

```python
uint8 -> unpack two int4 -> q - 8 -> gemm
```

更进阶的版本通常会继续加入：

```python
uint8 -> unpack -> q -> (q - zero_point) * scale -> gemm
```

常见工程变化包括：
- `scale` 按列存储（per-channel）
- `scale` 按 group 存储（group-wise）
- `zero_point` 显式存储，而不是固定写死为 8
- 反量化结果不落完整共享内存，而是更细粒度地喂给 MMA

### 3. 流水线深度

```python
# num_stages 选择
num_stages = 3  # 推荐 2-4

# 太少: 无法隐藏内存延迟
# 太多: 增加共享内存压力
```

## Part 5: 与标准 GEMM 的差异

### 内存访问模式

| 特性 | 标准 GEMM | 反量化 GEMM |
|------|----------|------------|
| A 矩阵加载 | 直接加载 | 直接加载 |
| B 矩阵加载 | 直接加载 | 加载 + 解包 + 反量化 |
| 计算核心 | T.gemm | T.gemm (相同) |
| 额外开销 | 无 | 反量化计算 |

### 数据流

```
标准 GEMM:
A (HBM) → A_shared → T.gemm → C_local → C (HBM)
B (HBM) → B_shared → T.gemm → C_local → C (HBM)

反量化 GEMM:
A (HBM) → A_shared → T.gemm → C_local → C (HBM)
B (HBM) → B_shared → B_dequantized → T.gemm
                      ↑
                   解包 + 教学版还原
```

如果扩展到真实量化 kernel，这个数据流通常还会插入 `scale/zero_point` 的读取和广播。
